# メール件名キーワード検知＋AI緊急度判定 設計書

作成日: 2026-07-04

## 目的

WEBメール（`webmail-rc.sixcore.ne.jp`、アカウント: `tenri@okashinodepart.net`）を定期的にチェックし、
件名に特定キーワード（「オンライン集計」「締め切り」「至急」など）を含む新着メールを検知する。
該当したメールについては、Claude APIでメール本文を解析し「緊急度」「返信要否」を自動判定してログに記録する。

通知はログファイルへの記録のみ（デスクトップ通知・LINE通知等は行わない）。

## 配置場所

`メール処理/`（既存の `メール処理/社長通信/` とは別用途。直下に配置）

## ファイル構成

| ファイル | 役割 |
|---|---|
| `config.json` | IMAP接続情報・キーワード一覧・`anthropic_api_key`（Gitに含めない） |
| `mail_check.py` | メイン処理 |
| `processed_uids.json` | 処理済みメールUIDの記録（重複検知防止、初回はベースライン登録のみ） |
| `mail_check_log.csv` | 検知結果ログ |
| `mail_check.log` | 実行ログ（接続エラー等） |
| `run_mail_check.bat` | タスクスケジューラから呼ぶ起動用バッチ |
| `check_mail_imap.py` | デバッグ用スクリプト（接続確認・件名一覧取得）。本番運用では使わない |

## config.json の構造

```json
{
  "imap_host": "okashinodepart.net",
  "imap_port": 143,
  "use_ssl": false,
  "user": "tenri@okashinodepart.net",
  "password": "ここにメールパスワードを入力してください",
  "keywords": ["オンライン集計", "締め切り", "至急"],
  "anthropic_api_key": "ここにClaude APIキーを入力してください"
}
```

実際の値（パスワード・APIキー）は `config.json` にのみ記載し、本ドキュメントやコードには書かない。

## 処理の流れ

1. `config.json` を読み込む
2. IMAP4（`okashinodepart.net:143`、SSLなし）でログインし `INBOX` を選択
3. `UID SEARCH ALL` で全メールのUIDを取得
4. `processed_uids.json` が存在しない（初回実行）場合:
   - 取得した全UIDをそのまま `processed_uids.json` に保存して終了（ログには何も書かない＝ベースライン化）
5. 2回目以降:
   - 取得UID一覧から `processed_uids.json` に記録済みのUIDを除外し、新着UIDのみ抽出
   - 新着UIDごとに件名・差出人・日時をヘッダから取得
   - 件名に `keywords` のいずれかが部分一致で含まれるか判定
   - 該当した場合のみ本文を取得し、Claude API（`claude-sonnet-5`）に渡して緊急度（高/中/低）・返信要否（要/不要）・判断理由を判定
     - API呼び出しが失敗した場合は「AI判定失敗」として扱い、キーワード検知自体は通常通りログに残す（処理は止めない）
   - 該当メールを `mail_check_log.csv` に1行追記（列: 実行日時, メール日時, 差出人, 件名, ヒットキーワード, 緊急度, 返信要否, AI判断理由）
   - 判定した新着UID（該当有無に関わらず）を `processed_uids.json` に追加保存

## AI緊急度判定

- SNSプロジェクト（`SNS/planner.py`）と同じ `anthropic` ライブラリ・`claude-sonnet-5` モデルを使用
- プロンプトにメール件名・本文を渡し、「緊急度（高/中/低）」「返信要否（要/不要）」「理由（1文）」を固定フォーマットのテキストで返させ、行単位でパースする
- キーワードに該当したメールのみAPIを呼び出す（コスト削減のため、全メールを毎回AI判定はしない）

## 実行タイミング

- Windowsタスクスケジューラで2時間ごと（9/11/13/15/17/19時）に実行
- タスク名は既存POSシステムの命名規則に合わせ `よしや_メールチェック` を想定

## エラー処理

- IMAP接続・認証に失敗した場合は例外を握りつぶさず `mail_check.log` に記録して終了する
- `processed_uids.json` は接続失敗時には更新しない（次回の定期実行で自然にリトライされる）
- Claude API呼び出し失敗時はログに「AI判定失敗」と記録し、キーワード検知結果自体は通常通り記録する

## テスト方針

- 本番の `config.json` を使う前に、`check_mail_imap.py`（デバッグ用）でIMAP接続確認・件名一覧取得を先に行う
- 初回実行時に既存メールがベースライン登録のみでログに残らないことを確認する
- キーワードに該当する新着メールで、AI判定結果を含めて `mail_check_log.csv` に正しく1行追記されることを確認する
